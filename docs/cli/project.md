# project — project

Project understanding: overview, full onboarding ("I just inherited this repo"), churn hotspots, changelog, init.

How to read this page:

- **Tool `name`** is the exact string for `tools/call`; **Workflow `name`** is `wf__<id>`.
- **Consent = yes** means the workflow changes state — over MCP it triggers an elicitation (or launch the server with `-y`).
- **LLM = mid/high** means the call sends content to a configured language model; `—` calls are fully local and free.
- If a call errors with "tool category disabled", enable it: `enable_category {"category":"project"}` (or launch with `--enable project`).

## Workflows

Workflows chain several tools into one action. Over MCP they are tools named `wf__<id>` (dots become underscores). Pass inputs in `arguments`.

| Workflow `name` | Needs | Consent | What it does |
|-----------------|-------|---------|-------------|
| `wf__project_changelog` | — | no | Update CHANGELOG.md |
| `wf__project_hotspots` | git | no | Churn hotspots: the files that change most (bug magnets) |
| `wf__project_init` | — | **yes** | Initialize a project |
| `wf__project_onboard` | — | no | Full onboarding: structure+profile+TODOs+scan+next steps |
| `wf__project_overview` | — | no | Project structure summary |

## Examples

Each line is the `tools/call` request a host sends over stdio (the `jsonrpc`/`id` envelope is elided):

```json
{"method":"tools/call","params":{"name":"wf__project_changelog","arguments":{}}}
```
