# harness — harness

The harness's self-management: effective config, version history, cost accounting, DB backup, update check.

How to read this page:

- **Tool `name`** is the exact string for `tools/call`; **Workflow `name`** is `wf__<id>`.
- **Consent = yes** means the workflow changes state — over MCP it triggers an elicitation (or launch the server with `-y`).
- **LLM = mid/high** means the call sends content to a configured language model; `—` calls are fully local and free.
- If a call errors with "tool category disabled", enable it: `enable_category {"category":"harness"}` (or launch with `--enable harness`).

## Workflows

Workflows chain several tools into one action. Over MCP they are tools named `wf__<id>` (dots become underscores). Pass inputs in `arguments`.

| Workflow `name` | Needs | Consent | What it does |
|-----------------|-------|---------|-------------|
| `wf__harness_backup` | — | no | Backup the DB |
| `wf__harness_config` | — | no | Show/edit config |
| `wf__harness_cost` | — | no | Token/cost accounting |
| `wf__harness_doctor` | — | no | Check dependencies |
| `wf__harness_history` | — | no | Version history |
| `wf__harness_update` | curl | **yes** | Check for updates |

## Examples
